# Optimizer Findings (Sonnet) — feat/environment-raw-management

## Summary

This branch adds two new capabilities: (1) a Python `environment-monitor` module that reads CBOR-encoded sensor records from `/persistent/*.RAW` files and downsamples temperature and humidity data into `bed_temp` and `freezer_temp` tables in `biometrics.db`; and (2) a set of new tRPC/REST endpoints — `environmentRouter` for querying those tables and `rawRouter` for listing, deleting, and inspecting disk usage of RAW files — plus a companion Next.js route handler (`/api/raw/[filename]`) that streams RAW file downloads. The install script is extended to deploy and activate the new service alongside the existing biometrics modules.

## Findings

### Finding 1: Duplicate migration file — one will never run

- **File**: `src/db/biometrics-migrations/0002_mushy_loki.sql` (entire file)
- **Severity**: 🔴 Critical
- **Category**: Correctness
- **Problem**: Two SQL files with identical content exist at `0002_fuzzy_strong_guy.sql` and `0002_mushy_loki.sql`. The migration journal (`meta/_journal.json`) only references `0002_fuzzy_strong_guy` as the canonical migration. `0002_mushy_loki.sql` is an orphaned duplicate that was generated by a second `drizzle-kit generate` run — it will never be applied by the Drizzle migrator, yet it sits in the migrations folder where future tooling may become confused about the migration sequence.
- **Suggested fix**: Delete `src/db/biometrics-migrations/0002_mushy_loki.sql`. It is not referenced in `meta/_journal.json` and is byte-for-byte identical to the canonical file. If the `0002_snapshot.json` was regenerated alongside it, verify the snapshot UUID matches the journal entry and remove the stale artefact.
- **Rationale**: Orphaned migration files are a maintenance hazard and can cause `drizzle-kit` to error with a "multiple files at same index" warning on the next schema change.

### Finding 2: `path.resolve(RAW_DIR)` path-traversal check is missing trailing separator

- **File**: `src/server/routers/raw.ts:68`, `app/api/raw/[filename]/route.ts:22`
- **Severity**: 🔴 Critical
- **Category**: Security
- **Problem**: Both locations check `resolved.startsWith(path.resolve(RAW_DIR))`. If `RAW_DATA_DIR` is set to a value like `/data`, a resolved path of `/data2/evil.RAW` would pass the check because `"/data2/evil.RAW".startsWith("/data")` is `true`. While the `SAFE_FILENAME` regex (which blocks path separators) makes this very difficult to exploit via the filename parameter directly, it does not protect against a misconfigured or attacker-influenced `RAW_DATA_DIR` environment value, and represents a latent bug waiting for a future refactor that relaxes `SAFE_FILENAME`.
- **Suggested fix**: Append a path separator to the canonical directory before the `startsWith` check:
  ```ts
  const canonicalDirWithSlash = path.resolve(RAW_DIR) + path.sep
  if (!resolved.startsWith(canonicalDirWithSlash) && resolved !== path.resolve(RAW_DIR)) {
    // reject
  }
  ```
  In `route.ts` the same fix applies to both the first check (line 22) and the `canonicalFile.startsWith(canonicalDir)` check (line 36).
- **Rationale**: The `startsWith` pattern without a trailing separator is a well-known path-traversal footgun documented in multiple CVEs. Fixing it now is cheap; the cost of a wrong assumption later is high.

### Finding 3: `raw.files` and `/api/raw/[filename]` have no authentication

- **File**: `src/server/routers/raw.ts:41-54`, `app/api/raw/[filename]/route.ts` (entire handler)
- **Severity**: 🟡 Major
- **Category**: Security
- **Problem**: `raw.files` (`protect: false`) and the Next.js download route have no authentication whatsoever. Any device reachable on the LAN can enumerate all RAW files and download their full binary contents. RAW files contain continuous multi-channel sensor recordings (bed temperature, humidity, pressure, HR, motion) that are high-sensitivity biometric data. `raw.deleteFile` does set `protect: true` in its OpenAPI meta, but this field is not currently enforced at the middleware level (the project uses `publicProcedure` everywhere — confirmed by the architecture docs and the fact that no other procedure uses `protect: true`). The flag is therefore cosmetic only.
- **Suggested fix**: At minimum, gate `/api/raw/[filename]` behind the same IP allowlist or session check that will be applied when authentication is added. Document explicitly in a comment that `protect: true` on `deleteFile` is aspirational and not currently enforced. When the auth middleware is implemented, ensure `deleteFile` is migrated to a real `protectedProcedure`.
- **Rationale**: RAW sensor data is the most sensitive biometric payload in the system; unauthenticated streaming of entire binary files is a higher risk than the existing unauthenticated JSON endpoints that already expose summarized biometric data.

### Finding 4: `getSummary` runs two sequential awaits on the same SQLite file — should be parallel

- **File**: `src/server/routers/environment.ts:190-218`
- **Severity**: 🟡 Major
- **Category**: Performance
- **Problem**: The `getSummary` handler issues two separate `biometricsDb.select(...).from(bedTemp)` and `biometricsDb.select(...).from(freezerTemp)` calls sequentially (one `await` after the other). SQLite in WAL mode supports concurrent readers. Because the Drizzle SQLite adapter over `better-sqlite3` is synchronous internally, the two queries will always run back-to-back even if wrapped in `Promise.all`, but the pattern is inconsistent with the same handler in `biometrics.ts` which already shows awareness of the issue. More importantly, if the project ever moves to a truly async driver (e.g. `@libsql/client`), sequential awaits will serialize unnecessarily.
- **Suggested fix**: Wrap both queries in `Promise.all`:
  ```ts
  const [bed, frz] = await Promise.all([
    biometricsDb.select({ ... }).from(bedTemp).where(...),
    biometricsDb.select({ ... }).from(freezerTemp).where(...),
  ])
  const [bedRow] = bed
  const [frzRow] = frz
  ```
- **Rationale**: This is an easy win for correctness-in-future and aligns with the project convention already used in other routers.

### Finding 5: All `output` schemas are `z.any()` — no type safety or contract enforcement

- **File**: `src/server/routers/environment.ts:34,74,109,146,177`, `src/server/routers/raw.ts:44,97`
- **Severity**: 🟡 Major
- **Category**: Type Safety
- **Problem**: Every procedure in both new routers uses `.output(z.any())`. This disables tRPC's end-to-end type inference for consumers and removes runtime output validation. The existing `biometrics.ts` router has the same pattern, but this branch adds 7 more `z.any()` output schemas in one go, widening the type-unsafe surface. Concrete output shapes are clearly defined in the router implementations — they could be typed today.
- **Suggested fix**: Define Zod output schemas inline or as named constants, e.g.:
  ```ts
  const bedTempRowSchema = z.object({
    id: z.number(),
    timestamp: z.date(),
    ambientTemp: z.number().nullable(),
    // ...
  })
  .output(z.array(bedTempRowSchema))
  ```
  For `raw.files` and `raw.diskUsage`, concrete schemas are straightforward.
- **Rationale**: Per the project architecture docs, full type safety is a core goal. `z.any()` on outputs defeats tRPC's type propagation to the frontend and removes the ability to catch response shape regressions at CI time.

### Finding 6: `deleteFile` active-file guard has a TOCTOU race condition

- **File**: `src/server/routers/raw.ts:73-80`
- **Severity**: 🟡 Major
- **Category**: Correctness
- **Problem**: The "newest file" guard calls `listRawFiles()` (which does a `readdir` + N `stat` calls) and then compares by filename. Between the `listRawFiles()` call and the `unlink()`, the hardware daemon may have rotated to a new RAW file, making the previously-guarded file stale and deleting it while data is still being written. Conversely, a new file might be created in that window, making the just-listed "newest" file no longer the active one. The guard also relies solely on mtime-sort ordering, which could give a wrong answer if multiple files share the same mtime.
- **Suggested fix**: The most robust approach is to identify the active file from the running daemon's state (e.g. a lock file or a PID file that indicates the current path) rather than inferring it from filesystem metadata. A simpler improvement is to reject deletion of any file modified within the last N seconds (e.g. 120s), regardless of sort order.
- **Rationale**: Deleting a file being actively written corrupts that file's data, which could cause the environment-monitor to crash or lose the current recording session.

### Finding 7: `df -B1` output parsing is fragile and platform-dependent

- **File**: `src/server/routers/raw.ts:101-107`
- **Severity**: 🟡 Major
- **Category**: Correctness
- **Problem**: The `diskUsage` handler parses `df -B1` output by splitting on whitespace and indexing columns `[1]`, `[2]`, `[3]` (Total, Used, Available). On macOS, `df` column order differs from Linux (macOS has `512-blocks, Used, Available, Capacity, iused, ifree, ...`); on some Linux distributions the output includes a header line that wraps. More importantly, if `RAW_DIR` contains a space, the path in column 0 may be split, shifting all subsequent column indices. Using `df -P` (POSIX output) is more portable and fixes the column-count issue, but still does not handle spaces in the path.
- **Suggested fix**: Use `df -P -B1 "$RAW_DIR"` for POSIX-portable output. Or better, use Node's `fs.statfs()` (available since Node 18.15) to avoid shelling out entirely:
  ```ts
  import { statfs } from 'node:fs/promises'
  const stats = await statfs(RAW_DIR)
  const totalBytes = stats.blocks * stats.bsize
  const availableBytes = stats.bavail * stats.bsize
  ```
- **Rationale**: The embedded device runs Linux, but the dev environment is macOS. Fragile `df` parsing will silently return zeros on macOS during development, hiding bugs, and could fail on unusual Linux configurations.

### Finding 8: Python monitor uses `check_same_thread=False` without enforcing single-thread access

- **File**: `modules/environment-monitor/main.py:81`
- **Severity**: 🟢 Minor
- **Category**: Correctness
- **Problem**: `sqlite3.connect(..., check_same_thread=False)` suppresses SQLite's built-in thread-safety guard. The module is single-threaded (main loop only), so this is functionally safe today. However, `report_health()` opens a separate connection to a *different* database (`SLEEPYPOD_DB`), while `db_conn` is used for `BIOMETRICS_DB`. The flag is set on `db_conn` but no thread actually accesses it from multiple threads. The flag gives a false impression that concurrent access is intended and could invite future bugs if a background thread is added (e.g. periodic health reporting).
- **Suggested fix**: Remove `check_same_thread=False` since the connection is never shared across threads. If concurrent access is ever needed later, use a connection per thread or a thread-safe queue.
- **Rationale**: Defensive correctness — suppress misleading flags that invite future unsafe patterns.

### Finding 9: `listRawFiles` in `raw.ts` uses `stat` instead of `lstat` — follows symlinks

- **File**: `src/server/routers/raw.ts:22`
- **Severity**: 🟢 Minor
- **Category**: Security
- **Problem**: `listRawFiles` calls `stat(path.join(RAW_DIR, name))` which follows symlinks. A symlink inside `RAW_DIR` pointing outside the directory would be listed and its `sizeBytes` would reflect the target file. The download route (`route.ts`) correctly rejects symlinks with `lstat`, but the listing router does not, so a user could discover information about files outside `RAW_DIR` via file size leakage. The `deleteFile` mutation also does not call `lstat` before `unlink`, which would follow a symlink and delete the symlink target.
- **Suggested fix**: Use `lstat` in `listRawFiles` and skip (or reject) entries that are symlinks. In `deleteFile`, add a `lstat` check analogous to the one in `route.ts`.
- **Rationale**: Consistent symlink rejection across the RAW file API surface. Defense in depth.

### Finding 10: `centiDegreesToC` and `centiPercentToPercent` accept `number` but callers pass `number | null`

- **File**: `src/lib/tempUtils.ts:33,37`, `src/server/routers/environment.ts:10-17`
- **Severity**: 🟢 Minor
- **Category**: Type Safety
- **Problem**: The new utility functions `centiDegreesToC(cd: number)` and `centiPercentToPercent(cp: number)` declare their parameters as `number`, not `number | null`. The wrapper functions `convertTemp` and `convertHumidity` correctly null-guard before calling them, but the type signature of the utilities themselves does not reflect that callers might inadvertently pass `null`. If TypeScript strict null checks are enabled and a future caller skips the null guard, the compiler will not catch the bug (it would require calling `centiDegreesToC(null)` which TypeScript would flag — but this is still a documentation/consistency gap vs the existing `toF`/`toC` functions).
- **Suggested fix**: The functions are fine as-is; just add JSDoc noting they expect non-null input, or add a runtime guard: `if (cd == null) return null` and change return type to `number | null`.
- **Rationale**: Minor type safety gap; documents intent explicitly.

### Finding 11: `environment-monitor` service has `ReadOnlyPaths=/persistent` overridden by `ReadWritePaths=/persistent/sleepypod-data`

- **File**: `modules/environment-monitor/sleepypod-environment-monitor.service:25-26`
- **Severity**: ⚪ Nit
- **Category**: Architecture
- **Problem**: The service specifies both `ReadOnlyPaths=/persistent` and `ReadWritePaths=/persistent/sleepypod-data`. In systemd, `ReadWritePaths` takes precedence over `ReadOnlyPaths` for overlapping paths, so the sub-path `/persistent/sleepypod-data` is effectively read-write (which is correct, since the module writes to `biometrics.db` there). This is the same pattern used by existing modules and works correctly — but it may be confusing to readers who expect `ReadOnlyPaths` to lock the entire tree. A comment would make the intent explicit. (Confirmed: same pattern in `piezo-processor` and `sleep-detector` services.)
- **Suggested fix**: Add a comment: `# ReadWritePaths overrides ReadOnlyPaths for the sub-path (systemd precedence rules)`.
- **Rationale**: Documentation clarity for operators reviewing sandboxing configuration.

### Finding 12: `report_health("down", ...)` on clean SIGTERM shutdown is misleading

- **File**: `modules/environment-monitor/main.py:196`
- **Severity**: ⚪ Nit
- **Category**: Correctness
- **Problem**: After the main loop exits normally (because `_shutdown` was set via SIGTERM/SIGINT), `report_health("down", "environment-monitor stopped")` is called. This marks the component as `"down"` in `system_health`. A systemd `Restart=always` service that is cleanly stopped by `systemctl stop` will show as `"down"` in the health table until the service restarts. Operators checking health during a planned deployment would see a false alarm.
- **Suggested fix**: Use a separate status like `"stopped"` for clean shutdowns vs `"down"` for unexpected exits. Or clear the health record on clean exit instead of setting status to `"down"`.
- **Rationale**: The existing `piezo-processor` has the same pattern (pre-existing), but since this is a new module it can be improved from the start.

### Finding 13: `getSummary` two-query approach — missing freezer `min`/`max` stats

- **File**: `src/server/routers/environment.ts:206-212`
- **Severity**: ⚪ Nit
- **Category**: Architecture
- **Problem**: The `getSummary` response includes `minAmbientTemp`/`maxAmbientTemp` for bed temperature, but the freezer temperature summary omits `min`/`max` entirely. Given that freezer temperature is the active cooling system's core metric, callers likely want range information (e.g. did heatsink temperature spike during the night?).
- **Suggested fix**: Add `minAmbient`, `maxAmbient`, `minHeatsink`, `maxHeatsink` to the freezer select and include them in the response shape for symmetry.
- **Rationale**: API surface inconsistency that will require a breaking change later if callers depend on the current shape.

### Finding 14: `wifiStatus` swallows all `nmcli` errors silently (pre-existing pattern, extended here)

- **File**: `src/server/routers/system.ts` (new `wifiStatus` procedure)
- **Severity**: 🟣 Pre-existing
- **Category**: Correctness
- **Problem**: The `catch` block for `nmcli` execution returns `{ connected: false, ssid: null, signal: null }` for any error — including genuine permission errors, unexpected nmcli output formats, or misparse. Callers cannot distinguish "WiFi interface absent" from "nmcli binary missing" from "parse error producing wrong answer". This is consistent with the silent-fallback pattern used elsewhere in `system.ts`.
- **Suggested fix**: Log the caught error at `debug` level so it appears in journald without alarming the user. Optionally, include an `error?: string` field in the output for diagnostic purposes.
- **Rationale**: Pre-existing pattern. Flagged for awareness.
